Module 1.1 Blog Post

Introduction.

Systems are the core of what we know today. They connect and work with practically everything, and it’d be hard to find anything that isn’t somehow connected. System Thinking is a widespread idea, which is the main thing we will be paying attention to, instead of breaking things down bit by bit in the tradition manor. With how the digital world is built and treated, understanding how all these factors work is a important thing to understand, in order to be able to work with it the best, we must first understand what it is and how the system works.

Defining System Thinking.

System Thinking is more of a broad idea vs a fact, it’s a way to think about how systems interact with each other and how they work with loops and how changes effect every end of a system. With how digital environments think and behave, every small bit of each induvial part and system are linked, every bit of hardware and software mesh to make the things we see all around us. All of the UI is also targeted towards people to grab attention, which is a underrated but yet another way it tries to mesh with everything.

Why System Thinking Is Crucial.

System Thinking is so crucial to us because of the relationship it has with everything around it. Everything and every single part has its purpose sure, but none of them can function as intended without them all interlinking and building off each other. It is also able to help diagnose issues and which section is failing, which is crucial to uptime and availability. In many ways, if there was no system thinking and all the engineering behind that, a lot of internet and other digital things would be much slower and much worse to work with, as errors and the smoothness we know today would vanish. And all of this isn’t just digital either, a lot of real world systems all use these factors in how they run, like marketing for example, a lot of the relationships are present, they are just targeted at consumers instead, and consumers keep business running, no consumers no money, which means everything goes under if everyone leaves.

What is Model Based System Engineering?

Model Based System Engineering or MBSE, is the different structures that make up a framework. They all share a common goal, to make a successful system (Holt, 32). A digital based Model is blueprint for how things need to function in order to work in the intended way, often having views as a core feature, which is all part of the larger framework. MBSE has evolved quite a bit, with 5 different stages, Document based, Document Centric, Model Enhanced, Model Centric, and Model Based System Engineering. All of these play such crucial parts when developing a model and really help break down each area to be clearer. I like to think of these stages as a big arrow from one end to the other, going from starting at a document or idea and shifting and evolving all the way into the MBSE to create a full picture, which I find to be really cool.

Why is it Important?

One of the biggest reasons it’s so important is it’s ability to break down each section of a system, and help make it less complicated. Whether that’s for researching, efficiently, trouble shooting, or other things, this is one of the key factors of MBSE. Another would be how it manages different systems and aligns everything, which makes everything run and work smoothly when operational. Being able to auto update and have a model that everything and every changes is seen helps streamline everything and makes life so much easier to work with when working with anything in the digital world, I think of it as a backbone in a way, as without it I feel like it would be much harder to work with or manage. Lastly, another factor I can think of would be on the human side of things. Humans help manage and run things too, and having everything streamlined helps prevent miscommunication and keeps shared goals in line and on time. In my mind, MBSE is what the peak of system Engineering itself has brought us, and I think it’s value is massively understated as a whole.

Summary.

In Summary, System Thinking and MBSE are two of the most crucial pieces of the digital world around us. Both need each other to function well, and both flourish when everything meshes well. Having such a well-planned out and efficient framework makes the flow of everything extremally smooth. Having a relationship like what System Thinking provides is a core function of how the digital world works. I think that with how much technology is evolving and how much the world is shifting that understanding all theses factors helps us evolve with it, and I find it fascinating to explore and dive into how things work and why.

Module 1.2 Blog Post

The System Development Life Cycle.

To define what the Systems Development Life Cycle is and what it means, we first must break it down in order to understand it. The best way I can think of to describe Systems development life cycle (or SDLC) is to picture it as a giant circle of life, or like a set blueprint that connects to itself over and over. Similar to in some religious places, how there is the idea that we as humans have a set cycle from birth to death, and then reincarnate back into birth again, completing a full circle, or in this case, a cycle. In this case however, it’s the system development side were focusing on for this blog.

The Life Cycle is built from several concepts, the Entity, the Stage, and the Life Cycle Model (Holt, 134). The Entity is where things start, and evolve and develop over time until they become the Stage. The Stage is where you can see checkpoints of where each major development and shift takes place. Lastly, the Life Cycle Model ties everything together and shows all the above concepts in action, specifically the behavioral side (Holt, 144). All of these mesh together beautifully and create the life cycle that is directly tied into SDLC.

Where it ties into the System Development side of things is best thought of in steps in my opinion. These steps include the base idea itself, Development, Implementation, Lifespan, and Retirement. Each of these 5 steps are key in how System Development works and functions, with each step having a set role and purpose. In many cases, even after something is retired, an Idea is formed based on the retired object, and a new life is breathed into it with improvements. In many cases I would say things get a 2.0 version, with major improvements while staying true to the core Ideas of the original inspiration. This helps tie into the Life Cycle just as a whole, it functions in very similar fashion.

The three types of Life Cycle Models

Now that we have broken down the Systems Development Life Cycle, it’s time to shift gears and talk about the different types of Life Cycle Models. There are three, the Linear, Iterative, and Incremental Life Cycle Models. Each are important in each way, and all have different features from each other, which give them all their own place. We will address each one in depth to further explore them.

Linear Life Cycle Model.

Key characteristics

The Linear Life Cycle Model is a super straightforward and simple Model. It follows a set sequence and pattern, often called a waterfall effect (Tipp, 2024). This is often used by having different stages that flow directly into one another in order. One of the key characteristics is that it’s very straightforward and easy to follow, because of the waterfall effect it has. This makes it very easy to use and implement, but can hinder flexibility because of the set path it’s on. Another would be how easy to approach this model is when it comes to using it, it’s so simple it’s much easier to understand and doesn’t appear as intimidating as other models.

Advantages

The great advantages of the Linear Life Cycle Model is as follows: Simple, straightforward, easy to approach, waterfall effect can be used well, understandable. All of these are worth highlighting to me as I think they are some of the top reasons to use the Linear Life Cycle Model.

Disadvantages

The disadvantages of the Linear Life Cycle Model are as follows: Lack of flexibility, changes are hard to imitate because of the waterfall effect, can take longer to work with, and has a higher risk of failure. This makes is feel rather outdated as a model to me, as while it seems useful, the time it could cost may not outweigh the advantages it poses to us to use.

Inerative Life Cycle Model.

Key characteristics

The Iterative Life Cycle Model is a much more flexible Model compared to the Linear Life Cycle Model we just talked about. This is because a lot of the phases and stages happen at the same time and work directly with each other and supporting each section, which leads to high success rates.

Advantages

There are several good advantages to the Iterative Life Cycle Model, High Flexibility, easy to work with, and lower risk levels. All of these make this a great Model for the workforce, as it’s super scalable, which is great for big projects.

Disadvantages

The downsides of the Iterative Life Cycle Model is that it is hard to implement, as there are a ton of moving parts, hard to keep in the constrains needed, and is unpredictable with both timelines and results. All of this is just due to how many moving parts are moving at the same time, which makes it challenging to work with sometimes unless properly planned for.

Incremental Life Cycle Model.

Key characteristics

The Incremental Life Cycle Model is yet another Model, but works differently compared to the two we have already looked at and broken down. The Incremental Life Cycle Model works by adding sections unto sections, building up a project or whatever bit by bit. This is helpful because it can break down each section to ensure success rates and put in the detail needed. Each section has it’s own phases to help keep a blueprint going, which I think is why it’s one of my favorites of the models because of how it functions and is structed.

Advantages

Advantages of the Incremental Life Cycle Model are: Easy to build and expand on, tighter control, easier to timeline a project because each section can be sectioned. This Model is one of the more practical use ones to me, and its modularity is why, I think that that is one of the biggest features of this model.

Disadvantages

There are some downsides however, it needs to be well implemented and planned out, or it has catastrophic risk potential. Another would be how if one section isn’t well done, the rest of the sections built upon that one will also have those negative aspects, which can be a major problem.

Why does the System Life Cycle matter for design, sustainability, and the evolution of digital systems?

The reason why all of this is so important is hard to nail down to just one concept or idea. The easiest way for me to explain it is to break it down into three sections, design, sustainability, and evolution of digital systems. This allows me to explain all the reasons why each one is so important, just for different reasons.

Design

The System Life Cycle matters because its design is key to how things run. Its design is also how it keeps itself on the path intended, without it, things could get more complicated then needed, it provides a blueprint design which really helps the entire process. How ideas are made, worked with, then transform into product or results is the beauty of it all.

Sustainability

One of the biggest factors of the life cycle is sustainability. That’s part of the circle path it follows, without it new things can’t be born from it. It’s also crucial that the model uses things that are easy to be updated or retired, because that keeps the flow going. This is important as it helps prevent a waste of time or energy having something waste away, when it can be reused, which in terms of greenness, is really important in this day and age. With how AI is, we need to try and have very green practices in order to counter that.

Evolution

Technology, which is the main thing when it comes to these models, are always evolving, being prepared for the constant change is a must. Nothing is set in stone in this day and age, and things are constantly tweaked and improved upon. This matters for the Life Cycle Model because it keeps it from getting outdated, which in the modern world, is one of the worst things to happen. This can also tie into sustainability, as outdated often means needing to be replaced, and if done wrong, it could just be a giant waste at the end of it all.

Summary

In Summary, The Life Cycle Model is one of the most important keys we have available to us. How it is designed and intended to be used is really fascinating to me, and I think it’s a great core to many things. Learning about the 3 major Life Cycle Models is really neat as well, as breaking down how each one works and how they differ from one another is very interesting. It seems to me that each one could be used in any situation, but understanding why you should use whatever model you select and why it should be used over the others is key. The benefits must outweigh the negatives, because they all have benefits and negatives, it’s just a process that needs to be well researched and well tailored for each use case. Thinking in these terms and having that mindset will only lead to more productivity, sustainability and well educated decision making, which I am all for. Building off of this information and these factors when it comes to these Models will make working with any of them a breeze in the future, which I think is a great bonus. That allows the impact potential to soar through the roof, it just depends on how you use the knowledge understanding this models give you.

Module 2.1 Blog Post

Introduction.

Human centered design is not a new concept, but it is one of the most important concepts in this modern and digital age. Human interaction might just be the most critical thing to keep our focus on, as with how the world is shifting, we are moving closer and closer to all Ai stuff, keeping Human things at the core will remain a challenge. This concept is key for those very reasons. That’s were Human Centered Design comes into play, its Human based at it’s core, but it really bends itself to adhere and appeal to the human nature. It’s design choices are made to speak to human nature, whether by colors, structure, it’s meant to feel alive. And with how much tech is evolving and reading online is becoming more prominent, this’ll help keep human traffic and keep peoples interest level up high.

Comparison of ISO 9241-210 and IDEO’s Field Guide.

ISO 9241-210 is a breakdown of how human centered design is built as a structure, while IDEO’s Field Guide is more human trait based and more guideline like vs the set structure that ISO 9241-210 possess. Both have their set uses, having a outline/framework is really nice to have something that is just solid in my opinion. But in all honesty, IDEO’s field guide is really interesting to me, as it has a lot of variance because of how the human nature works, anything empathy based is like a cheat sheet, it just works most of the time.

Examples of how perception and visual hierarchy affect usability.

Perception and Visual hierarchy are two different topics, but both apply in the same way and play key roles in keeping human interest level up. Perception is key because it is a make or break factor for many people. If something is poorly designed, it will be perceived poorly, which will lead to people clicking away. Visual hierarchy has a similar role, depending on how things are laid out, it can also be make or break. I can’t tell you the amount of times I’ve been on websites and hated how they’re laid out and went and found alternatives for what I was looking for. Both are dependent on each other, and if one fails they both fail, as the Human is most likely gone at this point. If something is unusable, or people think it’s unusable because it’s too complicated, it is a poor design in my opinion.

The four UX laws.

The four main UX laws are Jakob’s, Fitt’s, Hick’s, and Miller’s. While they are others, these are the main ones we’ll be diving into. All of them are based in various phycology and other human trait study’s, to try and get the best results possible if people adhere to these laws. While not laws in the tradition sense, they are more like general guidelines that help break things down, and they are great tools (Burgess, 2025).

Jakob's Law.

Jakob’s Law is when users spend a lot of time on the internet, in all sorts of different places, and they like it when the design you have is similar enough that they don’t have to think much to use what was made. This allows the design to stand out some, so people remember it, but keep the core functions similar to competitors, as this allows ease of use, which is crucial for keeping people interested.

Fitt's Law.

Fitt’s Law is based on object sizing when it comes to ease of use. The bigger an object, the easier it is for a user to see and apply to, as a smaller one is easier to misclick and harder to read. However, you can go to big, so finding that sweat spot between form and function is important.

Hick's Law.

Hick’s Law is all about simplicity. The more choices and options a user has, the harder for them to use it it’ll become because of that design choice. Overwhelming a user is a really big mistake, all it takes is one bad experience and they could never come back, so having your design function, but similarly, is crucial.

Miller's Law.

Miller’s Law is similar to and is kind of tied into Hick’s Law, as it has to do with overloading a user. The average person can hold about 7 items in their memory (Burgess, 2025), so if there is too much information to recall, it can cause several issues. It can overwhelm a user which is a bad idea, or it can lead to people making the wrong decision for their needs because they picked out bits and pieces of all the info, but not all of it because there simply was too much.

Summary.

In Summary, there really is a lot that goes into Human Centered Design. I personally think it’s one of if not the core function when it comes to designing. In a world so full of AI and how many websites and data points are targeted to have all the info you need or only some of it, I think remembering our core principles is super important. While the Laws aren’t binding so to speak, there are really useful guidelines to help keep human interaction at the center of interest. I can’t imagine how the world would be if/when we lose that in the future, so I think it’s really important to keep fighting to keep Humans centered. ISO 9241-210 and IDEO’s Field Guide both provided some really good points based on structure vs human sided emotions and factors, I think that is the framework that all of this is built on. And while perception and visual hierarchy effect usability so much, I think it’s important to understand how to work with that to help everything tie in together well. Ultimately, I think there are a lot of ways to keep Human Centered Design relevant for now and the future, but I do think it’s crucial we use all those factors and take everything into consideration, as it can become a dying art very quickly. Touching those Human factors like emotions and beauty of design I think will go a long way, as that can really be touching for people, and if that works, you’ll keep them interested and invested in what you have. While functionality is also important and not the main topic, that can work super well with Human Centered Design, and is a core part for a reason. With both of these combined, I believe there can still be a lot of success in a quickly changing and almost intimating digital world and society.
